# Phase 1 — UI Design Contract: Foundation + Trips

> Visual and interaction contract for Phase 1. Generated by gsd-ui-researcher.
> This phase establishes the base design language for all subsequent phases.
> Sources: PROJECT.md (Clean & native constraint), REQUIREMENTS.md (FOUND-01..03, TRIP-01..06), ARCHITECTURE.md (NavigationStack, sheet patterns), Apple HIG iOS 17.
>
> **Revision 2026-04-19** — reconciled after smoke-test feedback. Five changes:
> 1. Root shell is a `TabView` (Trips + Settings) wrapping the NavigationStack.
> 2. Color scheme is **dark-only** — `.preferredColorScheme(.dark)` locked at the app root.
> 3. Trip Detail body uses **3 cards** (Documents + Packing half-width over full-width Activities), replacing the segmented `Picker`.
> 4. Trip name is **optional** — empty input persists as `"Untitled Trip"`.
> 5. Destination reorder affordance is list-row drag only — no `EditButton` in TripEditSheet.

## Spacing Scale

SwiftUI uses points (pt), not pixels. On standard (1x) displays 1pt = 1px; on Retina (2x/3x) it scales automatically. The scale below maps to multiples of 4pt, matching the Apple 8-point grid.

| Token | Value | SwiftUI Usage |
|-------|-------|---------------|
| xs | 4pt | SF Symbol insets, badge padding |
| sm | 8pt | Within-row gaps (icon to label), inline padding |
| md | 16pt | Default List row horizontal padding, Form section inset |
| lg | 24pt | Section header spacing, sheet title top margin |
| xl | 32pt | Empty state icon bottom margin |
| 2xl | 48pt | Empty state vertical centering offset |
| 3xl | 64pt | Not used in Phase 1 |

**Exceptions:**
- Minimum touch target: 44pt × 44pt (Apple HIG mandatory). Applies to toolbar buttons, sheet dismiss buttons, and any tappable row.
- Destination tag chip horizontal padding: 12pt (`md` ÷ 4 × 3 — one step above `sm`) — provides adequate breathing room for chip text at narrow screen widths while remaining Dynamic Type-safe; aligns with the 4pt grid.
- NavigationStack large title clearance: handled automatically by SwiftUI `.navigationBarTitleDisplayMode(.large)` — do not override.

## Typography

SwiftUI text styles map to SF Pro automatically and respond to Dynamic Type. Declare roles by text style name, not raw point size. Raw sizes listed are the default (non-accessibility) sizes for reference only.

| Role | SwiftUI Text Style | Default Size | Weight | Line Spacing |
|------|-------------------|-------------|--------|--------------|
| Display | `.largeTitle` | 34pt | Regular (400) | System default (1.2 ratio) |
| Heading | `.title2` | 22pt | Semibold (600) | System default (1.2 ratio) |
| Body | `.body` | 17pt | Regular (400) | System default (1.5 ratio) |
| Label / Secondary | `.subheadline` | 15pt | Regular (400) | System default (1.4 ratio) |

**Usage rules:**
- Display (`.largeTitle`): NavigationStack large title only (trip list screen title "Trips"). Not used inline.
- Heading (`.title2` semibold): Trip name in TripDetailView header. Sheet titles ("New Trip", "Edit Trip").
- Body (`.body`): Trip list row primary label (trip name). Form field input text. Date range display.
- Label / Secondary (`.subheadline`): Trip list row secondary line (date range, destination count). Destination tag chip text. Form field labels rendered by `Form` automatically.

**Dynamic Type:** All roles use system text styles — Dynamic Type scaling is automatic. Do not hard-code point sizes anywhere in Phase 1 implementation.

**Font weight constraint:** Exactly 2 weights in use — Regular (400) and Semibold (600). No Bold (700), no Medium (500), no Light (300).

## Color

All colors use iOS semantic system colors resolved in **Dark Mode only**. The app applies `.preferredColorScheme(.dark)` at the root `WindowGroup` scene, so semantic colors always resolve to their dark-appearance values regardless of the user's system setting. No hex values are hardcoded — semantic names are referenced instead.

| Role | SwiftUI Color | Semantic Meaning | Usage |
|------|--------------|-----------------|-------|
| Dominant (60%) | `.systemBackground` | Primary surface | App background, List background, Sheet background |
| Secondary (30%) | `.secondarySystemBackground` | Elevated surface | Grouped List rows, Form sections, card-like containers |
| Accent (10%) | `.accentColor` (system blue) | Interactive / brand | See reserved list below |
| Destructive | `.red` (system red) | Destructive-only | Delete trip button, delete confirmation |

**Accent reserved for — explicit list (not "all interactive elements"):**
1. "New Trip" toolbar button (plus icon)
2. "Create Trip" / "Save Changes" button (sheet nav bar, right side)
3. Tappable destination chip underline (if implemented as links)
4. Date picker selection highlight (handled automatically by SwiftUI `DatePicker`)

Accent is NOT used for: list row backgrounds, trip name text, section headers, non-interactive icons, or navigation bar chrome.

**Secondary semantic color — Labels:**
- `.primary` — trip name, form field values, sheet headings
- `.secondary` — date range secondary line, destination count, form field placeholder text, empty state body copy
- These are SwiftUI's built-in label colors, not custom values.

**Outdoor readability note (from platform_context):** Dark Mode is the only supported appearance in v1. `.systemBackground` resolves to near-black (#000000) and card surfaces (`.secondarySystemBackground`) to #1C1C1E, providing the highest contrast for on-trip outdoor use. No custom color overrides that could reduce contrast. A future Settings toggle (Phase 6+) may re-enable system-driven appearance — until then, light-mode resolution paths are not tested.

## Copywriting Contract

### Primary CTA

| Element | Copy |
|---------|------|
| Primary CTA — Trip list toolbar | "New Trip" (SF Symbol: `plus`) |
| Primary CTA — New Trip sheet confirm | "Create Trip" |
| Primary CTA — Edit Trip sheet confirm | "Save Changes" |
| Primary CTA — Add Destination inline | "Add Destination" (SF Symbol: `plus.circle`) |

### Empty State — Trip List

Shown when `@Query` returns zero Trip records. Displayed centered in the List area.

| Element | Copy |
|---------|------|
| Empty state heading | "No Trips Yet" |
| Empty state body | "Create your first trip to get started." |
| Empty state CTA | "New Trip" (same button as toolbar — do not add a second button) |

**Visual:** SF Symbol `airplane.departure` at `.font(.system(size: 56))` in `.secondary` color. 32pt gap below icon, heading below that, body 8pt below heading.

### Trip Detail Shell — Section Cards

The trip detail screen shows Documents, Packing, and Activities as three stacked cards in Phase 1 as a scaffold. Layout:

- Top row: `HStack` with two half-width cards — **Documents** (left) and **Packing** (right).
- Below: one full-width **Activities** card (taller `minHeight` so it anchors the screen).
- Cards share a `SectionCard` view: SF Symbol + title (`.headline`) header row, body copy (`.subheadline` secondary), rounded 16pt corner, `.secondarySystemBackground` fill, 16pt internal padding, 12pt horizontal inter-card gap, 16pt vertical stack gap.

Card copy:

| Card | SF Symbol | Title | Body |
|------|-----------|-------|------|
| Documents | `doc.text` | "Documents" | "Documents will appear here." |
| Packing | `checklist` | "Packing" | "Your packing list will appear here." |
| Activities | `calendar` | "Activities" | "Your itinerary will appear here." |

Bodies are temporary and will be replaced in Phases 2–4 with real content inside the same card shape. Keep them neutral — no promises about "coming soon."

### Root Shell — TabView

The app root is a `TabView` with two tabs. Settings is a placeholder until Phase 6.

| Tab | SF Symbol | Title | Phase 1 content |
|-----|-----------|-------|-----------------|
| Trips | `airplane` | "Trips" | `NavigationStack` → `TripListView` → `TripDetailView` |
| Settings | `gearshape` | "Settings" | `NavigationStack` → single list row: "Settings coming soon." |

### Error States

| Situation | Copy |
|-----------|------|
| Trip save fails (SwiftData error) | "Couldn't save trip. Please try again." (presented as `.alert`) |
| End date before start date | "End date must be on or after the start date." (inline validation below date pickers) |

**Note:** Trip name is optional. Empty input is persisted as `"Untitled Trip"`; no inline error is shown.

### Destructive Actions

Phase 1 has exactly one destructive action: Delete Trip.

| Action | Trigger | Confirmation approach |
|--------|---------|----------------------|
| Delete Trip | Swipe-left on trip list row → "Delete" button (red, system destructive) | `.confirmationDialog` (action sheet on iPhone): Title: "Delete \"{trip name}\"?" / Message: "This will also delete all documents, packing items, and activities for this trip." / Buttons: "Delete Trip" (destructive role) + "Cancel" (cancel role) |

**Note:** No inline undo. Deletion is permanent in v1. The confirmation dialog copy makes this explicit via the cascading consequence message.

## Component Inventory

Components used in Phase 1 only. All are stock SwiftUI — no custom components introduced in this phase unless listed.

| Screen | Component | SwiftUI API | Accessibility |
|--------|-----------|-------------|---------------|
| Trip List | Sorted list of trips | `List` with `@Query(sort:)` | — |
| Trip List | Swipe-to-delete | `.swipeActions(edge: .trailing)` | — |
| Trip List | Add button | `ToolbarItem(placement: .navigationBarTrailing)` with `Button` | `.accessibilityLabel("New Trip")` — required; the button renders a `plus` symbol with no visible text label |
| New Trip Sheet | Input form | `NavigationStack` + `Form` presented as `.sheet` | — |
| New Trip Sheet | Trip name field | `TextField("Trip name", text: $name)` | — |
| New Trip Sheet | Date range | Two `DatePicker` controls (Start Date, End Date) in `.compact` display mode | — |
| New Trip Sheet | Destination list | `List` with inline `TextField` per destination row | — |
| New Trip Sheet | Add destination | `Button("Add Destination")` at bottom of destination list | — |
| New Trip Sheet | Reorder destinations | `.onMove` on the destination list (drag-handle only — no EditButton toolbar toggle) | — |
| New Trip Sheet | Remove destination | `.onDelete` or swipe action on destination row | — |
| Trip Detail | Header | `VStack` with trip name (`.title2` semibold), date range (`.subheadline` secondary), destination chips | — |
| Trip Detail | Section cards | 3× `SectionCard` (Documents + Packing `HStack`, Activities full-width below) inside a `ScrollView` | Each card exposes its title + body to VoiceOver as a grouped element |
| Root | Tab shell | `TabView` with `Trips` + `Settings` items | Tabs self-label via `.tabItem { Label(...) }` |
| Delete Confirmation | Destructive dialog | `.confirmationDialog` | — |

**Trip Detail layout decision — Phase 1 (revised 2026-04-19):** Use three stacked `SectionCard` views (Documents + Packing half-width in an `HStack`, Activities full-width below). The earlier segmented `Picker` was replaced because it hid two of three sections behind a tap and clashed with the upcoming root `TabView`. Cards let each section grow independently in Phases 2–4 and give Activities the visual weight it needs as the densest content.

**Destination chips in Trip Detail header:** Display as a horizontal `ScrollView` + `HStack` of `Text` views styled as rounded rectangle labels. Use `.background(Color.secondarySystemBackground)` + `.clipShape(Capsule())`. Horizontal padding: 12pt (see Spacing Scale exceptions). No tap action in Phase 1 (edit goes through Edit Trip sheet).

## Interaction Contracts

### Navigation Flow

```
ContentView
  └── TabView
        ├── Tab "Trips" (airplane)
        │     └── NavigationStack (path: [AppDestination])
        │           ├── TripListView                      ← large title "Trips"
        │           │     └── .sheet(isPresented: $showNewTrip)
        │           │           └── TripEditSheet(.create) ← NavigationStack inside sheet
        │           └── TripDetailView(tripID:)            ← pushed on row tap
        │                 └── .sheet(isPresented: $showEdit)
        │                       └── TripEditSheet(.edit(trip))
        └── Tab "Settings" (gearshape)
              └── NavigationStack
                    └── SettingsView                       ← placeholder (Phase 6)
```

### Gesture Contracts

| Gesture | Element | Result |
|---------|---------|--------|
| Tap | Trip list row | Push TripDetailView |
| Swipe left | Trip list row | Reveal red "Delete" button |
| Tap "Delete" | Swipe action | Show `.confirmationDialog` |
| Tap "Delete Trip" in dialog | Confirmation | Delete trip + children, pop if on detail |
| Tap toolbar "New Trip" | Trip list | Present NewTripSheet as `.sheet` |
| Tap "Edit" | Trip detail toolbar | Present EditTripSheet as `.sheet` |
| Tap "Cancel" | Sheet nav bar | Dismiss sheet, no save |
| Tap "Create Trip" / "Save Changes" | Sheet nav bar | Validate → save → dismiss |

### Validation Rules

| Field | Rule | Feedback |
|-------|------|----------|
| Trip name | Optional — empty input persists as `"Untitled Trip"` | None; Save remains enabled on empty |
| Start date | Required | DatePicker prevents nil; default to today |
| End date | Required; must be >= start date | Inline error below end date picker |
| Destinations | Optional (zero or more) | No validation; user can have a trip with no explicit destination |
| Destination name | Non-empty if row exists | Remove row on empty+blur, or disable Save with inline error |

### Sheet Dismiss Behavior

- "Cancel" in a New Trip sheet with unsaved data: dismiss without prompt (no data to lose — field is not persisted until Save).
- "Cancel" in an Edit Trip sheet: restore original values (do not mutate the model until Save is confirmed).
- Swipe-down to dismiss sheet: same behavior as Cancel.

## Phase 1 Locked Decisions (Inherited by All Later Phases)

These decisions are made here and must not be revisited without a design contract amendment:

| Decision | Value | Rationale |
|----------|-------|-----------|
| Accent color | System blue (`.accentColor`) | Defers to user's iOS accent color preference; respects accessibility; no brand lock-in in v1 |
| Font | SF Pro via system text styles only | No custom fonts; Dynamic Type support automatic; matches "clean & native" mandate |
| Color mode | **Dark only** — `.preferredColorScheme(.dark)` at root | Outdoor readability is the primary use case; one appearance halves the visual QA surface. Re-opening to system appearance becomes a Phase 6+ Settings concern |
| Spacing unit | 4pt base, 8pt grid | Matches Apple HIG; aligns with SwiftUI `padding()` defaults |
| Touch target floor | 44pt × 44pt | Apple HIG minimum; never go below this for any tappable element |
| Destructive pattern | `.confirmationDialog` with cascading-consequence message | One consistent pattern for all destructive actions across all phases |
| Sheet structure | `NavigationStack` inside `.sheet` | Required to show navigation bar buttons (Cancel / Save) inside a sheet in SwiftUI |
| Trip Detail layout | 3 `SectionCard`s (Documents + Packing half-width, Activities full-width) | Replaces segmented `Picker`; each section stays visible and gets its own growth runway in Phases 2–4 |
| Root shell | `TabView` (Trips + Settings) wrapping each tab's own `NavigationStack` | Settings tab already claims the slot Phase 6 (DOC-08) will populate, avoiding a later shell refactor |
| Trip name | Optional — empty persists as `"Untitled Trip"` | Friction removal; users can rename later via Edit Trip |
| Destination reorder affordance | List-row drag only (no `EditButton`) | "Edit" label in a Create sheet was confusing; drag handles appear automatically with `.onMove` |
