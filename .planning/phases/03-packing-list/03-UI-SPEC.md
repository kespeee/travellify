# Phase 3 — UI Design Contract: Packing List

> Visual and interaction contract for Phase 3. Generated by gsd-ui-researcher.
> Inherits every locked decision from Phase 1 and Phase 2 UI-SPECs (dark theme, SF Pro system text
> styles, system colors, 4pt spacing grid, `.accentColor` for interactive affordances,
> `.confirmationDialog` / `.alert` for destructive actions, long-press `.contextMenu` for headers).
> Sources: CONTEXT.md D19–D39 (user decisions), Phase 1 + Phase 2 UI-SPECs (established conventions),
> REQUIREMENTS.md PACK-01..PACK-07, CLAUDE.md, codebase scan of TripDetailView.swift /
> EmptyDocumentsView.swift / DocumentListView.swift.

This phase does NOT redefine spacing, typography, or color. It inherits the Phase 1/2 contract
verbatim and only specifies (a) new surfaces introduced by the Packing feature and (b)
copy/interaction contracts unique to Phase 3.

## Color

Inherited verbatim from Phase 1 UI-SPEC. Dark-only via `.preferredColorScheme(.dark)` at app root.

| Role | SwiftUI Color | Phase 3 Usage |
|------|---------------|---------------|
| Dominant (60%) | `.systemBackground` | `PackingListView` list background |
| Secondary (30%) | `.secondarySystemBackground` | `SectionCard` Packing card in `TripDetailView`; list section background (default `List` grouped style) |
| Accent (10%) | `.accentColor` (system blue inherited, `.tint` in code) | See reserved list below |
| Destructive | `.red` | Trailing swipe delete button background; "Delete" button inside category `.confirmationDialog` |

**Accent reserved for — Phase 3 additions (additive to Phase 1/2 list):**
1. Leading swipe check-off button background (green tint — see swipeActions note below).
2. `checkmark` SF Symbol on the leading swipe action button.
3. `ProgressView(value:)` fill color on the trip-level progress bar.
4. Rename alert "Save" button (SwiftUI `.alert` default-action tint — inherited pattern).
5. Category context menu "Rename" item's `pencil` SF Symbol.
6. `plus.circle` SF Symbol on the "Add item" placeholder row (`.accentColor`).
7. `plus.circle` SF Symbol on the "Add category" row (`.accentColor`).

> **Note on leading swipe tint:** `.swipeActions(edge: .leading)` with `Button` having no explicit
> role uses `.tint(.green)` to differentiate the positive check-off action from the destructive
> trailing delete. This is the only surface in Phase 3 that uses `.green` instead of `.accentColor`
> (system blue) — intentional Mail idiom divergence for a positive toggle action.

**Accent is NOT used for:** item row text, item row checked state (`.secondary`), section header
text, the `X/Y` counter label, the progress "X / Y packed" text, or the `SectionCard` card icon in
TripDetail (uses `.tint` inherited from Phase 1 card pattern).

**Checked state color:**
- Item name: `.primary` → `.secondary` (`.foregroundStyle(item.isChecked ? .secondary : .primary)`)
- Strikethrough: applied via `.strikethrough(item.isChecked, color: .secondary)`
- Item row does NOT change its background color on check — no background flash or color shift.

### Empty State — Packing List (zero categories)

Shown when the trip's `packingCategories` relationship is empty. Centered in the List area. Identical structural pattern to `EmptyDocumentsView` / `TripEmptyState`.

| Element | Copy / Rendering |
|---------|-----------------|
| Icon | SF Symbol `checklist` at `.font(.system(size: 56))`, `.secondary` color, `.accessibilityHidden(true)` |
| Heading | "No Categories Yet" (`.title2.weight(.semibold)`, `.primary`) |
| Body | "Tap "Add category" below to start building your packing list." (`.subheadline`, `.secondary`, `.multilineTextAlignment(.center)`) |
| Spacing | 32pt below icon, 8pt between heading and body (matches Phase 1/2 empty state rhythm) |
| Layout | `VStack(spacing: 0)` with `.padding(.bottom, 32)` after icon, `.padding(.bottom, 8)` after heading; `.padding(.horizontal, 32)` on outer `VStack`; `.frame(maxWidth: .infinity, maxHeight: .infinity)` |

**The "Add category" row remains visible at the bottom of the list even when the empty state is shown.** The empty state body copy points to that row — do not add a duplicate `Button` inside the empty state body. The "Add category" row is the single discoverable action entry point (consistent with Phase 2 D15 pattern — no CTA duplication).

**Combined VoiceOver label:** `"No categories yet. Tap Add category below to start building your packing list."`

### Progress Header Row

Shown at the top of the list, above the first section. Present whenever categories exist (replaces empty state). Scrolls away with the list content (not sticky) — per D24.

| Element | Copy / Rendering |
|---------|-----------------|
| Label | `"\(checkedCount) / \(totalCount) packed"` (`.subheadline`, `.secondary`); 0 / 0 shows "0 / 0 packed" |
| Progress bar | `ProgressView(value: Double(checkedCount), total: Double(max(totalCount, 1)))` — `max(…, 1)` guards against division-by-zero when totalCount is 0; `.tint(.accentColor)`; `.progressViewStyle(.linear)` |
| Layout | `VStack(alignment: .leading, spacing: 4)` containing label above bar; full-width within `List` row insets; row is NOT tappable |
| Accessibility | `.accessibilityElement(children: .ignore)` + `.accessibilityLabel("\(checkedCount) of \(totalCount) items packed")` + `.accessibilityValue("\(progressPercent)%")` |

### Inline Add Item Row (per section)

Appears as the last row within each section, below all existing items.

| State | Rendering |
|-------|-----------|
| Default (not editing) | `HStack { Image(systemName: "plus.circle").foregroundStyle(.accentColor) Text("+ Add item").font(.subheadline).foregroundStyle(.secondary) }` |
| Focused (after tap) | Row transforms: `TextField("Item name", text: $newItemName)` with `@FocusState` set to `.focused`; `.font(.body)`; `.submitLabel(.done)`; on submit: if non-empty → insert item + clear field + re-focus (rapid multi-add per D30); if empty → dismiss focus |
| Return / submit | Non-empty: insert `PackingItem` with `name = trimmed(newItemName)`, append sortOrder, re-focus for next entry. Empty: clear field, dismiss focus. |

### Add Category Row (bottom of list)

Appears as a single row below the last section.

| State | Rendering |
|-------|-----------|
| Tappable row | `HStack { Image(systemName: "plus.circle").foregroundStyle(.accentColor) Text("Add category").font(.subheadline).foregroundStyle(.secondary) }` |

| Action | Trigger | Confirmation |
|--------|---------|--------------|

- `titleVisibility: .visible`

### Error States

| Situation | Copy | Presentation |
|-----------|------|--------------|
| SwiftData save fails on item insert | "Couldn't add item. Please try again." | `.alert` on `PackingListView`, single "OK" button |
| SwiftData save fails on item rename | "Couldn't rename. Please try again." | `.alert` on `PackingListView`, single "OK" button |
| SwiftData save fails on item delete (swipe) | "Couldn't delete item. Please try again." | `.alert` on `PackingListView`, single "OK" button |
| SwiftData save fails on category insert | "Couldn't add category. Please try again." | `.alert` on `PackingListView`, single "OK" button |
| SwiftData save fails on category rename | "Couldn't rename category. Please try again." | `.alert` on `PackingListView`, single "OK" button |
| SwiftData save fails on category delete | "Couldn't delete category. Please try again." | `.alert` on `PackingListView`, single "OK" button |

### Destructive Actions

Phase 3 introduces two destructive surfaces.

| Action | Trigger | Confirmation |
|--------|---------|--------------|
| Delete item | Trailing swipe on item row → full swipe or "Delete" button (`.swipeActions(edge: .trailing, allowsFullSwipe: true)`, `Button(role: .destructive)`) | **No confirmation dialog.** Per D27, individual item delete is non-confirmed (cost of mistake is low — user retypes a name). Consistent with SwiftUI `List` destructive-swipe convention. |
| Delete category | Long-press section header → `.contextMenu` → "Delete" (destructive role) → `.confirmationDialog` (see copy above) | `.confirmationDialog` required because cascade deletes all items in the category (irreversible bulk action). |

## Interaction Contracts

### Navigation Flow

```
TripListView (Phase 1)
  └── push → TripDetailView (Phase 1)
        ├── Documents SectionCard (Phase 2)
        │     └── tap → push AppDestination.documentList(tripID)
        └── Packing SectionCard (Phase 3 wire-up)
              └── tap → push AppDestination.packingList(tripID)
                    └── PackingListView (Phase 3)
                          ├── Progress header row (scrolls with list)
                          ├── Section per category
                          │     ├── Section header (long-press)
                          │     │     └── .contextMenu
                          │     │           ├── "Rename" → .alert with TextField
                          │     │           └── "Delete" → .confirmationDialog
                          │     ├── Item rows
                          │     │     ├── single tap     → inline rename (TextField replaces Text)
                          │     │     ├── leading swipe  → toggle isChecked (+ sensoryFeedback)
                          │     │     └── trailing swipe → delete item (no confirmation)
                          │     └── "+ Add item" row
                          │           └── tap → @FocusState focuses inline TextField
                          └── "Add category" row
                                └── tap → .alert with TextField
```

### Gesture Contracts

| Gesture | Element | Result |
|---------|---------|--------|
| Tap | Packing SectionCard (TripDetail) | Push `AppDestination.packingList(tripID)` |
| Tap | Item row (existing item name) | Enter inline rename mode (`@FocusState` → TextField focused, current name selected) |
| Leading swipe (right) | Item row | Toggle `item.isChecked`; `.sensoryFeedback(.success)` fires; item stays in place |
| Full leading swipe | Item row | Same as above (allowsFullSwipe: true) |
| Trailing swipe (left) | Item row | Delete item with no confirmation (per D27); SwiftData save; on failure → error alert |
| Full trailing swipe | Item row | Same as above (allowsFullSwipe: true) |
| Long-press | Section header | Present `.contextMenu` (Rename + Delete) |
| Tap | Context menu "Rename" | Dismiss context menu, present rename `.alert` prefilled with `category.name` |
| Tap | Context menu "Delete" | Dismiss context menu, present delete `.confirmationDialog` |
| Tap | Rename alert "Save" | Trim draft; if non-empty, save `category.name`, dismiss alert |
| Tap | Rename alert "Cancel" / outside | Discard draft, dismiss alert |
| Tap | Delete dialog "Delete" | `context.delete(category)` (cascades to items via D22); save; row disappears |
| Tap | Delete dialog "Cancel" / outside | Dismiss dialog, no state change |
| Tap | "+ Add item" row | Focus inline `TextField`; keyboard appears |
| Return (non-empty name) | Inline add-item TextField | Insert `PackingItem`; clear field; keep focus for next entry (D30) |
| Return (empty) | Inline add-item TextField | Dismiss focus without inserting |
| Tap outside | Inline add-item TextField | Dismiss focus without inserting |
| Submit rename (inline) | Inline rename TextField on existing item | Trim draft; if non-empty, save `item.name`, dismiss focus; if empty, revert to original, dismiss focus |
| Tap | "Add category" row | Present new-category `.alert` |
| Tap | New-category alert "Add" | Insert `PackingCategory`; dismiss alert |
| Tap | New-category alert "Cancel" | Discard, dismiss alert |
| Long-press + drag | Item row | Begin drag; drag onto another section → reassign `item.category`, recompute `sortOrder` at destination |
| Drop | Section area | Accept item from same list; persist reassignment |

### Validation Rules

| Field | Rule | Feedback |
|-------|------|----------|
| New category name | Trimmed whitespace; must be non-empty | "Add" button disabled while trimmed input is empty |
| Rename category name | Trimmed whitespace; must be non-empty | "Save" button disabled while trimmed draft is empty |
| Inline add item name | Trimmed whitespace; must be non-empty to insert | Return with empty field dismisses focus (no item inserted) |
| Inline rename item name | Trimmed whitespace; must be non-empty to save | Empty submit reverts to original name and dismisses focus |
| Category name max length | Soft cap — no enforced limit; `.lineLimit(1)` truncation in header | — |
| Item name max length | Soft cap — no enforced limit; `.lineLimit(1)` truncation in row | — |

### Focus State Contracts

The `PackingListView` manages two distinct focus domains via `@FocusState`:

1. **Add-item focus per section:** `@FocusState private var addItemFocus: PersistentIdentifier?` — set to `category.persistentModelID` on tap of "+ Add item" row; cleared on return (empty) or tap-outside. Only one add-item field is focused at a time.

2. **Rename-item focus per row:** `@FocusState private var renameItemFocus: PersistentIdentifier?` — set to `item.persistentModelID` on tap of an existing item row; cleared on submit or tap-outside. Only one rename field is active at a time.

These two focus domains are mutually exclusive: gaining focus in one clears the other (SwiftUI `@FocusState` switching is automatic).

### sensoryFeedback Contracts

| Trigger | API | When Fires |
|---------|-----|-----------|
| Item checked | `.sensoryFeedback(.success, trigger: item.isChecked)` | On `isChecked` change to `true` |
| Item unchecked | `.sensoryFeedback(.success, trigger: item.isChecked)` | On `isChecked` change to `false` |

Both directions use `.success` feedback — symmetric check/uncheck idiom per D29. The trigger is `item.isChecked` (any change fires feedback), which covers both toggling directions.
