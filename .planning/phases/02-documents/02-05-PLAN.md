## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user rename input → displayName (NOT fileRelativePath) | User-typed text must NEVER touch the on-disk path |
| delete action → FileStorage.remove / FileStorage.removeTripFolder | Path construction must go through FileStorage validators |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-08 (carry-over from Plan 02-02) | Tampering | Rename Save closure | mitigate | Code path explicitly writes to `displayName` only; `fileRelativePath` is never mentioned in the closure. Acceptance criterion `grep -c "doc.fileRelativePath =" ... returns 0` enforces this statically. |
| T-02-20 | Information Disclosure | Delete logging | mitigate | `Logger.fileStorage.error` uses `privacy: .private` for `doc.id`; only `error.localizedDescription` is public (POSIX strings; no PII). |
| T-02-21 | Tampering | Trip cascade folder removal with manipulated UUID | mitigate | `trip.id` is a `UUID` (Phase 1 `var id: UUID = UUID()`), converted via `uuidString` which returns only `[0-9A-F-]`. `FileStorage.validateComponent` additionally rejects `/`, `..`, leading `.`. Defense in depth. |
| T-02-22 | DoS/Data Integrity | Folder removal fails mid-run | mitigate | `try?` swallows folder removal; orphan files are tolerable (storage cost only; no correctness impact). Model delete has already saved successfully. |
| T-02-23 | Tampering/Elevation | Rename saves fileRelativePath mutation (regression) | mitigate | grep-gate enforced in acceptance criterion. Any future edit writing to `fileRelativePath` inside DocumentListView must justify bypassing this gate. |
</threat_model>
